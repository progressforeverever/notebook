# ConcurrentHashMap 1.8

Java 7 中 ConcurrentHashMap 的存储结构如上图，ConcurrnetHashMap 由很多个 Segment 组合，而每一个 Segment 是一个类似于 HashMap 的结构，所以每一个 HashMap 的内部可以进行扩容。但是 Segment 的个数一旦初始化就不能改变，默认 Segment 的个数是 16 个，你也可以认为 ConcurrentHashMap 默认支持最多 16 个线程并发。

# 1. 存储结构

可以发现 Java8 的 ConcurrentHashMap 相对于 Java7 来说变化比较大，不再是之前的 Segment 数组 + HashEntry 数组 + 链表，而是 Node 数组 + 链表 / 红黑树。当冲突链表达到一定长度时，链表会转换成红黑树。

无参构造中调用了有参构造，传入了三个参数的默认值，他们的值是。

接着看下这个有参构造函数的内部实现逻辑。

  1. 必要参数校验。
  2. 校验并发级别 concurrencyLevel 大小，如果大于最大值，重置为最大值。无参构造默认值是 16.
  3. 寻找并发级别 concurrencyLevel 之上最近的 2 的幂次方值，作为初始化容量大小，默认是 16。
  4. 记录 segmentShift 偏移量，这个值为【容量 = 2 的 N 次方】中的 N，在后面 Put 时计算位置时会用到。默认是 32 - sshift = 28.
  5. 记录 segmentMask，默认是 ssize - 1 = 16 -1 = 15.
  6. 初始化 segments[0]，默认大小为 2，负载因子 0.75，扩容阀值是 2*0.75=1.5，插入第二个值时才会进行扩容。

# 2. 初始化 initTable

/**
 * Initializes table, using the size recorded in sizeCtl.
 */
private final Node<K,V>[] initTable() {
    Node<K,V>[] tab; int sc;
    while ((tab = table) == null || tab.length == 0) {
        //　如果 sizeCtl < 0 ,说明另外的线程执行CAS 成功，正在进行初始化。
        if ((sc = sizeCtl) < 0)
            // 让出 CPU 使用权
            Thread.yield(); // lost initialization race; just spin
        else if (U.compareAndSwapInt(this, SIZECTL, sc, -1)) {
            try {
                if ((tab = table) == null || tab.length == 0) {
                    int n = (sc > 0) ? sc : DEFAULT_CAPACITY;
                    @SuppressWarnings("unchecked")
                    Node<K,V>[] nt = (Node<K,V>[])new Node<?,?>[n];
                    table = tab = nt;
                    sc = n - (n >>> 2);
                }
            } finally {
                sizeCtl = sc;
            }
            break;
        }
    }
    return tab;
}

从源码中可以发现 ConcurrentHashMap 的初始化是通过自旋和 CAS 操作完成的。里面需要注意的是变量 sizeCtl ，它的值决定着当前的初始化状态。

  2. -N 说明有 N-1 个线程正在进行扩容

  4. >0 表示 table 扩容的阈值，如果 table 已经初始化。

# 3. put

直接过一遍 put 源码。

public V put(K key, V value) {
    return putVal(key, value, false);
}

/** Implementation for put and putIfAbsent */
final V putVal(K key, V value, boolean onlyIfAbsent) {
    // key 和 value 不能为空
    if (key == null || value == null) throw new NullPointerException();
    int hash = spread(key.hashCode());
    int binCount = 0;
    for (Node<K,V>[] tab = table;;) {
        // f = 目标位置元素
        Node<K,V> f; int n, i, fh;// fh 后面存放目标位置的元素 hash 值
        if (tab == null || (n = tab.length) == 0)
            // 数组桶为空，初始化数组桶（自旋+CAS)
            tab = initTable();
        else if ((f = tabAt(tab, i = (n - 1) & hash)) == null) {
            // 桶内为空，CAS 放入，不加锁，成功了就直接 break 跳出
            if (casTabAt(tab, i, null,new Node<K,V>(hash, key, value, null)))
                break;  // no lock when adding to empty bin
        }
        else if ((fh = f.hash) == MOVED)
            tab = helpTransfer(tab, f);
        else {
            V oldVal = null;
            // 使用 synchronized 加锁加入节点
            synchronized (f) {
                if (tabAt(tab, i) == f) {
                    // 说明是链表
                    if (fh >= 0) {
                        binCount = 1;
                        // 循环加入新的或者覆盖节点
                        for (Node<K,V> e = f;; ++binCount) {
                            K ek;
                            if (e.hash == hash &&
                                ((ek = e.key) == key ||
                                 (ek != null && key.equals(ek)))) {
                                oldVal = e.val;
                                if (!onlyIfAbsent)
                                    e.val = value;
                                break;
                            }
                            Node<K,V> pred = e;
                            if ((e = e.next) == null) {
                                pred.next = new Node<K,V>(hash, key,
                                                          value, null);
                                break;
                            }
                        }
                    }
                    else if (f instanceof TreeBin) {
                        // 红黑树
                        Node<K,V> p;
                        binCount = 2;
                        if ((p = ((TreeBin<K,V>)f).putTreeVal(hash, key,
                                                       value)) != null) {
                            oldVal = p.val;
                            if (!onlyIfAbsent)
                                p.val = value;
                        }
                    }
                }
            }
            if (binCount != 0) {
                if (binCount >= TREEIFY_THRESHOLD)
                    treeifyBin(tab, i);
                if (oldVal != null)
                    return oldVal;
                break;
            }
        }
    }
    addCount(1L, binCount);
    return null;
}

# 4. get

get 流程比较简单，直接过一遍源码。

public V get(Object key) {
    Node<K,V>[] tab; Node<K,V> e, p; int n, eh; K ek;
    // key 所在的 hash 位置
    int h = spread(key.hashCode());
    if ((tab = table) != null && (n = tab.length) > 0 &&
        (e = tabAt(tab, (n - 1) & h)) != null) {
        // 如果指定位置元素存在，头结点hash值相同
        if ((eh = e.hash) == h) {
            if ((ek = e.key) == key || (ek != null && key.equals(ek)))
                // key hash 值相等，key值相同，直接返回元素 value
                return e.val;
        }
        else if (eh < 0)
            // 头结点hash值小于0，说明正在扩容或者是红黑树，find查找
            return (p = e.find(h, key)) != null ? p.val : null;
        while ((e = e.next) != null) {
            // 是链表，遍历查找
            if (e.hash == h &&
                ((ek = e.key) == key || (ek != null && key.equals(ek))))
                return e.val;
        }
    }
    return null;
}

# 总结

Java7 中 ConcurrentHashMap 使用的分段锁，也就是每一个 Segment 上同时只有一个线程可以操作，每一个 Segment 都是一个类似 HashMap 数组的结构，它可以扩容，它的冲突会转化为链表。但是 Segment 的个数一但初始化就不能改变。

Java8 中的 ConcurrentHashMap 使用的 Synchronized 锁加 CAS 的机制**。结构也由 Java7 中的 **Segment 数组 + HashEntry 数组 + 链表进化成了 Node 数组 + 链表 / 红黑树，Node 是类似于一个 HashEntry 的结构。它的冲突再达到一定大小时会转化成红黑树，在冲突小于一定数量时又退回链表。

有些同学可能对 Synchronized 的性能存在疑问，其实 Synchronized 锁自从引入锁升级策略后，性能不再是问题，有兴趣的同学可以自己了解下 Synchronized 的锁升级。

ConcurrentHashMap 是 Java 中并发访问安全的哈希表实现，它允许多个线程同时读取和修改映射表，而不需要额外的同步措施（比如锁），因此在多线程环境下具有较好的性能。

下面是关于 ConcurrentHashMap 的详细介绍：

  1. 线程安全性： ConcurrentHashMap 提供了比 Hashtable 和通过同步包装的 HashMap 更高的并发性能。它采用了一种称为分段锁（Segmented Locking）的机制，将整个数据分割成一定数量的段（Segment），每个段相当于一个小的 HashMap，并且每个段都有自己的锁。这样，当一个线程修改其中一个段时，不会影响到其他段，从而实现了更细粒度的并发控制。

ConcurrentHashMap 的线程安全性是通过几个关键的实现策略来保证的，主要包括分段锁（Segmented Locking）、CAS 操作（Compare and Swap）和 volatile 变量等。下面具体介绍 ConcurrentHashMap 的线程安全实现：

  1. 分段锁（Segmented Locking）： ConcurrentHashMap 内部将数据结构分成多个段（Segments），每个段相当于一个小的 HashMap，具有独立的锁。默认情况下，ConcurrentHashMap 包含 16 个段，因此可以支持最多 16 个线程同时进行写操作，提高了并发性能。

  2. 获取锁的过程： 当需要进行写操作时，首先根据键的哈希值定位到对应的段，然后只锁定这个段，而不是整个 ConcurrentHashMap。这样，其他段的操作不会被阻塞，允许多个线程同时读取不同的段。

  3. 保证并发访问安全的操作：

    • 读操作（get）：读操作是非阻塞的，可以同时进行。每个段的读操作不需要加锁，只需要通过 volatile 变量读取数据，保证可见性。
    • 写操作（put、remove 等）：写操作会先获取相应段的锁，确保同一时刻只有一个线程能够修改这个段的数据。其他线程可以同时修改其他段，从而提高并发性。

  4. CAS 操作（Compare and Swap）： ConcurrentHashMap 使用 CAS 操作来保证更新操作的原子性。CAS 是一种乐观锁的实现方式，在执行更新操作时，先比较当前值和期望值是否相等，如果相等则进行更新，否则重试或者放弃。

  5. 扩容机制： 当某个段的元素数量超过阈值时，会触发扩容操作，但这个扩容只会影响到需要扩容的那个段，而不会影响其他段。在扩容过程中，其他段的操作不会被阻塞。

  6. 保证可见性的方式： ConcurrentHashMap 使用 volatile 变量来保证数据的可见性。在多线程环境下，一个线程对共享变量的修改会立即对其他线程可见，因为其他线程读取该变量时会通过 volatile 变量获取最新值。

总体来说，ConcurrentHashMap 通过分段锁、CAS 操作和 volatile 变量等技术实现了高效的并发访问。它在多线程环境中能够提供较好的性能和线程安全保障，适用于高并发读写的场景，如并发缓存、数据统计等应用场景。

  1. 数据结构： ConcurrentHashMap 内部采用数组 + 链表 / 红黑树的数据结构来存储键值对。数组的每个元素都是一个链表或红黑树的头节点，用于存放哈希冲突的键值对。
  2. 扩容机制： ConcurrentHashMap 在初始化时会创建一定数量的段（默认为 16），随着元素的不断增加，每个段会根据元素数量动态调整其容量，从而控制每个段中存放的键值对数量。当某个段的大小超过阈值时，会触发扩容操作，但这个扩容只影响到需要扩容的那个段，而不会影响其他段，因此并不会阻塞其他线程的访问。

读操作： ConcurrentHashMap 的读操作是非阻塞的，允许多个线程同时进行读取操作，因为不需要加锁。在读操作中，线程只会通过 volatile 变量来读取数据，因此不会引起线程阻塞。

ConcurrentHashMap 的读操作是针对多线程并发访问设计的，具有高效的并发性能。读操作主要涉及定位元素、获取元素和保证可见性等步骤。下面是 ConcurrentHashMap 读操作的具体过程：

  1. 定位元素的段（Segment）： 首先，根据键的哈希值确定元素所在的段。ConcurrentHashMap 内部使用哈希值的高位来确定元素应该放置在哪个段中。因此，根据元素的哈希值，可以直接确定需要操作的段。

  2. 定位元素的桶（Bucket）： 在确定了元素所在的段后，根据元素的哈希值定位到段内的具体桶（Bucket）。桶是存放键值对的地方，每个桶可以包含多个键值对，可能是一个链表或者红黑树结构。

  3. 遍历桶内的链表或红黑树： 如果桶内有多个键值对（可能形成链表或红黑树），则需要遍历桶内的数据结构来找到对应的键值对。具体的查找方式会根据键的哈希值进行匹配。

  4. 获取元素： 一旦定位到了正确的键值对，即可获取对应的值。在获取值之前，通常需要通过 volatile 变量来保证数据的可见性，确保读取到的值是最新的值。

  5. 保证可见性： ConcurrentHashMap 使用 volatile 变量来保证对元素的读操作的可见性。这意味着当一个线程获取到了某个元素的值之后，其他线程可以立即看到这个最新的值，而不会出现数据不一致的情况。

总体来说，ConcurrentHashMap 的读操作是非阻塞的，并发性能较高。

写操作： ConcurrentHashMap 的写操作是安全的，它通过分段锁来控制并发访问。当一个线程需要修改数据时，只需要锁定相应的段，而不是整个数据结构，从而提高了并发性能。

  1. 定位元素的段（Segment）： 首先，根据键的哈希值确定元素所在的段。ConcurrentHashMap 使用哈希值的高位来确定元素应该放置在哪个段中。
  2. 获取段的锁： 写操作需要先获取对应段的锁（Segment-level lock）。ConcurrentHashMap 中每个段都有一个独立的锁，用于控制并发访问。获取段的锁可以保证同一时刻只有一个线程能够对该段进行修改操作，从而确保并发写操作的线程安全性。
  3. 定位元素的桶（Bucket）： 在确定了元素所在的段后，根据元素的哈希值定位到段内的具体桶（Bucket）。
  4. 插入、更新或删除操作：
    • 插入操作：将新的键值对插入到对应的桶中。如果插入后桶内的键值对数量超过阈值（通常是8个），可能会触发链表转换为红黑树的操作。
    • 更新操作：根据键查找到对应的键值对，然后更新其值。
    • 删除操作：根据键查找到对应的键值对，然后将其从桶中删除。
  5. 释放段的锁： 写操作完成后，需要释放对应段的锁，允许其他线程访问同一段进行读或写操作。
  6. 扩容操作： 如果写操作导致某个段内的元素数量超过阈值，可能会触发扩容操作。扩容过程会涉及到重新计算哈希值并迁移元素到新的段中，确保整体的并发性能和负载因子。

总体来说，ConcurrentHashMap 在多线程环境下提供了高效的并发访问支持，适用于需要高并发读写操作的场景，例如在高并发的 web 服务器中用作全局的数据缓存，或者用于存储共享的状态信息等。

# volatitle

volatile 是 Java 中的关键字，用于修饰变量，主要用于确保变量的可见性和禁止指令重排序，从而保证线程之间对变量的操作是正确的和可预测的。

volatile 的特性：

  1. 可见性（Visibility）： 当一个变量被 volatile 关键字修饰时，如果一个线程修改了这个变量的值，那么其他线程能够立即看到这个变量的最新值。这是因为 volatile 变量的修改会立即被写回主内存，而读取操作时会从主内存中重新获取最新值，而不是使用线程本地的缓存值。

  2. 禁止指令重排序（Prevents Instruction Reordering）： volatile 变量的读写操作会插入内存屏障（Memory Barrier），防止指令重排序，确保对 volatile 变量的操作顺序与代码中的顺序一致。

示例说明：

假设有一个共享的 boolean 类型的变量 flag，用于标识某个任务是否完成。多个线程需要检查这个标志位，并根据标志位的状态来做相应的处理。

public class VolatileExample {
    private volatile boolean flag = false;

    public void startTask() {
        // 模拟任务执行
        System.out.println("Task started...");
        // 任务执行完成后设置标志位为true
        flag = true;
    }

    public void checkFlag() {
        while (!flag) {
            // 如果标志位为false，继续等待
        }
        System.out.println("Flag is now true. Task completed.");
    }

    public static void main(String[] args) {
        VolatileExample example = new VolatileExample();

        // 启动一个线程执行任务
        new Thread(() -> {
            example.startTask();
        }).start();

        // 主线程检查标志位
        example.checkFlag();
    }
}

在上面的例子中，flag 被声明为 volatile boolean 类型。任务执行完成后，flag 被设置为 true。在 checkFlag() 方法中，使用一个循环来不断检查 flag 的值，直到 flag 变为 true，表示任务已经完成。因为 flag 是 volatile 的，所以主线程在检查 flag 时能够立即看到任务线程修改后的最新值，从而正确地判断任务是否完成，避免了线程之间的可见性问题。

总之，volatile 可以用于确保多线程之间对变量的可见性，但需要注意它并不能保证原子性，因此在涉及到复合操作的场景中，仍然需要使用锁或其他并发控制机制来确保线程安全。
